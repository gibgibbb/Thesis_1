## 1. Introduction

### 1.1 Logistic Regression in this Thesis

This thesis evaluates two machine learning models — Logistic Regression (LR) and Random Forest (RF) — as ignition probability estimators within a Cellular Automata (CA) fire spread simulation for Lapu-Lapu City, Cebu. This chapter focuses on the LR component, while the RF component is documented in [reference Kent's chapter].

LR was selected as a baseline model for three reasons:
- **Interpretability:** LR's linear coefficients allow direct inspection of how each environmental factor contributes to ignition probability — useful for both scientific understanding and stakeholder communication
- **Computational efficiency:** LR is orders of magnitude faster to train and apply than tree ensembles, making it suitable for real-time fire spread prediction on resource-constrained hardware
- **Methodological rigor:** comparing a simple linear baseline against a more complex non-linear model (RF) lets us quantify how much architectural complexity is needed for this prediction task

### 1.2 Research Question

The thesis-level research question is: **can a CA+ML hybrid system, trained on synthetic data generated by physics-grounded fire spread rules, predict the spatial extent of real fire incidents in Lapu-Lapu City?**

Within that, this chapter answers two sub-questions:
- How well does Logistic Regression perform as the ignition probability estimator?
- What features and hyperparameter configurations are necessary to extract maximum performance from a linear model on this problem?

## 2. Methodology

### 2.1 Feature Set

The model receives the following features per candidate cell at each simulation timestep:

| # | Feature | Source | Range |
|---|---------|--------|-------|
| 1 | slope_risk | Raster (DEM-derived) | 0.0 – 1.0 |
| 2 | proximity_risk | Raster (distance-to-roads) | 0.0 – 1.0 |
| 3 | building_presence | Raster (footprints) | 0 or 1 |
| 4 | material_risk | Raster (material category lookup) | 0.0 – 0.95 |
| 5 | material_class | Raster (categorical 0–5) | integer |
| 6 | wind_speed | Simulation config | 5.0 – 25.0 km/h |
| 7 | wind_sin | Derived from wind direction | -1.0 – 1.0 |
| 8 | wind_cos | Derived from wind direction | -1.0 – 1.0 |
| 9 | neighbor_burning_count | Dynamic (uniform 3×3 convolution) | 1 – 8 |
| 10 | composite_flammability | building_presence × material_risk | 0.0 – 0.95 |
| 11 | wind_weighted_score | Dynamic (directional 3×3 convolution, normalized) | 0.0 – 1.0 |

Feature 11 (`wind_weighted_score`) was added during this investigation after discovering that its absence created an information gap between the simulation physics and the ML input — see Section 3.3 below.

**Susceptibility filter:** training data records only cells where `neighbor_burning_count > 0`. This matches what the CA engine evaluates at runtime — cells with no burning neighbors are not candidates for ignition. Including those cells in training would make `neighbor_burning_count` a trivial separator and produce misleadingly high accuracy.

### 2.2 Model Architecture

The LR model is wrapped in an sklearn Pipeline:

```python
Pipeline([
    ("scaler", StandardScaler()),
    ("lr", LogisticRegression(max_iter=2000, random_state=42, ...)),
])
```

The `StandardScaler` normalizes features to zero mean and unit variance before passing to LR. This is necessary because LR is sensitive to feature magnitudes (e.g., `wind_speed` ranges 5–25 while `building_presence` is 0–1). The Pipeline wraps both transforms into a single object exposing `predict_proba()`, satisfying the contract required by the CA engine for runtime model integration.

**Mathematical formulation:** LR computes a linear combination of features and applies the logistic (sigmoid) function:

```
score(x) = w₁·f₁ + w₂·f₂ + ... + wₙ·fₙ + b
P(ignite | x) = 1 / (1 + exp(-score(x)))
```

This formulation can capture monotonic, additive relationships between features and ignition probability but cannot represent feature interactions natively — a limitation we explicitly investigated in Section 3.4.

### 2.3 Synthetic Training Data

Real fire perimeter data is unavailable for Lapu-Lapu City — the Bureau of Fire Protection provided only ~28 incident records consisting of text addresses without spatial extents. Training data was therefore generated synthetically by running the CA simulation across diverse wind configurations.

**Generation parameters:**
- 5 wind speeds: 5, 10, 15, 20, 25 km/h
- 8 wind directions: N, NE, E, SE, S, SW, W, NW (45° increments)
- 5 random seeds per (speed, direction) combination → different ignition points
- 100 timesteps per scenario
- 10 ignition points per scenario, sampled from burnable cells
- Total: **200 simulation runs → 21,079 cell-timestep observations → 1,095 ignitions (5.19% positive rate)**

Each row in the training dataset records the features at time `t` and the label `Ignited ∈ {0, 1}` indicating whether the cell transitioned to the IGNITED state at time `t+1`.

**Acknowledged circularity:** the model trains on data generated by the CA's own ignition formula and is later inserted back into the CA at inference time. This is a known limitation of synthetic-data ML pipelines. We mitigate it by (a) grounding the CA in published fire physics (Alexandridis et al. 2011, Gao et al. 2008), (b) framing the model as learning emergent aggregate spread behavior rather than the exact formula, and (c) validating the final system on real fire incident data (Section 3.5). The methodology is transferable to real fire data when available.

### 2.4 Per-Cell Evaluation

For diagnostic investigation, we evaluate the model on a held-out 20% test split (stratified by `Ignited`) using:
- **Confusion matrix** (TN, FP, FN, TP)
- **Precision, Recall, F1-score** (with threshold optimization in [0.10, 0.90, step 0.05])
- **AUC-ROC**
- **Jaccard index**

These per-cell metrics are diagnostic — they measure how well the model has learned to reproduce the CA's ignition rule on synthetic data. The thesis-defining metric is the spatial validation in Section 3.5, not these per-cell numbers.

### 2.5 Hyperparameter Optimization

Logistic Regression's regularization hyperparameter `C` governs the bias-variance tradeoff (Hastie et al. 2009): higher values risk overfitting while lower values constrain model capacity. Since the optimal value is dataset-dependent and cannot be determined analytically, we employed Bayesian hyperparameter optimization via Optuna (Akiba et al. 2019) using the Tree-structured Parzen Estimator (TPE) sampler with stratified 5-fold cross-validation (Kohavi 1995) to systematically identify the configuration that maximizes F1-score on held-out validation folds.

**Search space:**
- `penalty`: `{l1, l2, elasticnet}`
- `solver`: chosen to be compatible with `penalty` (`liblinear`, `lbfgs`, `saga`)
- `C`: continuous in `[10⁻⁴, 10²]`, log-uniform sampling
- `class_weight`: `balanced` or custom positive weight in `[1.0, 20.0]`
- `l1_ratio`: `[0.0, 1.0]` if `penalty == elasticnet`

**Trial budget:** 100 trials per configuration. Convergence was tracked by recording the best trial number; in all runs the best trial was found before trial 60, indicating the search space was adequately explored.

**Test set isolation:** the 20% held-out test set was never seen by Optuna. The CV score reported by Optuna is the optimization target; the test score is reported separately as the honest final metric. The CV–test gap quantifies any overfitting introduced by hyperparameter selection.

### 2.6 Feature Engineering

Because LR cannot natively capture feature interactions, we manually constructed seven physics-informed interaction features:

| Engineered Feature | Formula | Justification |
|--------------------|---------|---------------|
| slope_x_neighbors | slope_risk × neighbor_burning_count | Fire travels uphill faster (Alexandridis 2011) |
| wind_x_neighbors | wind_speed × neighbor_burning_count | Wind fans flames from neighbors (Gao 2008) |
| slope_x_building | slope_risk × building_presence | Buildings on slopes more vulnerable to uphill fire |
| slope_x_wind | slope_risk × wind_speed | Terrain–wind channeling effect |
| proximity_x_building | proximity_risk × building_presence | Wildland-urban interface risk |
| proximity_x_neighbors | proximity_risk × neighbor_burning_count | Edge accessibility to fire |
| neighbors_squared | neighbor_burning_count² | Nonlinear convergent-heat effect |

These were evaluated as a separate 16/17-feature variant (depending on whether `wind_weighted_score` was present).

### 2.7 Spatial Validation

Per the thesis proposal, the primary validation compares the simulation output against historical fire incident data in Lapu-Lapu City (2019–present). The simulation is initialized with an ignition point matching the real fire's known location (converted from EPSG:32651 GPS coordinates to grid indices via the affine transform), runs forward for `max_timesteps`, and produces a `final_state` raster with cell states {1: NON_BURNABLE, 2: NOT_YET_BURNING, 3: IGNITED, 4: BLAZING, 5: EXTINGUISHED}.

Cells in states {3, 4, 5} are treated as "fire" predictions and compared to a binary ground truth raster constructed by manually digitizing the real fire's burned area in QGIS using post-fire Google satellite imagery. (BFP did not provide official burned area polygons; this is a documented data limitation.)

The validation metrics are the same as the per-cell metrics, but computed over all ~38 million cells in the raster:
- Precision, Recall (primary), F1-score, AUC-ROC, Jaccard

**Per the proposal, recall is the primary metric** with target ≥ 0.80, since false negatives (missing real fire spread) are more costly than false positives (slightly over-predicting fire extent) in an early-warning context.

## 3. Results

### 3.1 Wind Fix Discovery

**Initial finding:** during exploratory analysis, we discovered that the CA engine's `step()` method used a constant wind multiplier (`1.0 + wind_weight = 1.2`) regardless of the configured wind speed or direction. The three wind features (`wind_speed`, `wind_sin`, `wind_cos`) were recorded in training data but had no effect on simulation behavior. LR coefficients for all three were exactly 0.000, confirming the features carried no signal.

**Fix:** we replaced the flat multiplier with a directional 3×3 wind kernel (`_compute_wind_kernel()`) that weights neighboring cells by their alignment with the wind direction:
- Upwind neighbors (fire traveling WITH the wind toward the candidate cell) receive weights > 1.0
- Downwind neighbors receive weights as low as 0.05
- The effect is scaled by `speed_factor = speed_kmh / 10.0`

This formulation is grounded in the canonical CA fire models of Alexandridis et al. (2011) and Gao et al. (2008), which both treat wind as a directional multiplier on neighbor-driven spread probability.

**Validation:** after the fix, retraining LR produced non-zero wind coefficients (`wind_speed: +0.096`, `wind_sin: +0.060`, `wind_cos: +0.027`), confirming the simulation now propagates fire differently under different wind configurations.

### 3.2 Per-Cell 8-Variant Comparison (Diagnostic)

We systematically evaluated LR across 8 configurations to characterize the model's performance and identify the primary performance limiter.

| # | Variant | Features | F1 | Recall | Precision | AUC-ROC |
|---|---------|:--------:|:--:|:------:|:---------:|:-------:|
| 1 | 9feat, defaults | 9 | 0.133 | 0.470 | 0.077 | 0.630 |
| 2 | 9feat, Optuna | 9 | 0.129 | 0.379 | 0.078 | 0.629 |
| 3 | 9 + engineered, defaults | 16 | 0.135 | 0.489 | 0.078 | 0.631 |
| 4 | 9 + engineered, Optuna | 16 | 0.134 | 0.489 | 0.078 | 0.629 |
| 5 | 10feat (+ wind_weighted), defaults | 10 | 0.162 | 0.306 | 0.110 | 0.656 |
| 6 | 10feat, Optuna | 10 | 0.164 | 0.237 | 0.125 | 0.656 |
| 7 | 10 + engineered, defaults | 17 | 0.165 | 0.320 | 0.111 | 0.658 |
| 8 | 10 + engineered, Optuna | 17 | 0.165 | 0.320 | 0.111 | 0.658 |

Cross-validation–test gaps remained below 0.02 across all variants, indicating no overfitting through hyperparameter selection.

### 3.3 Information Gap Discovery

The most important diagnostic finding was an **information asymmetry between the CA engine and the ML model**:

- The CA engine internally computes a directional `wind_weighted_score = convolve(blazing, wind_kernel) / max_kernel_sum` to decide ignition probability
- However, the ML model received only `neighbor_burning_count` — a uniform integer count produced by a different convolution

In other words, the CA knew which neighbors were burning and weighted them by wind alignment, while the ML model only knew how many neighbors were burning. A cell with two upwind burning neighbors (highly dangerous) appeared identical in feature space to a cell with two downwind burning neighbors (much safer).

**Fix:** we added `wind_weighted_score` as a tenth feature, computed using the same directional kernel the CA uses. The feature was added consistently in the data generator, the feature pipeline, and the runtime prediction path.

**Effect:** F1 improved from 0.133 → 0.162 (+22%) on per-cell evaluation, with `wind_weighted_score` becoming the second-strongest predictor (coefficient +0.301) immediately after `proximity_risk` (+0.318). Notably, this single feature contributed more performance improvement than all seven hand-engineered interactions combined (+0.029 vs +0.002).

This finding generalizes beyond LR. Both LR and RF received identical features. The information gap likely limited both models. Whether RF benefits similarly from `wind_weighted_score` is being evaluated separately by the RF author (Kent) and is not within the scope of this chapter. However, see §3.6 for an important caveat: the per-cell improvement from `wind_weighted_score` does not translate proportionally to spatial validation, suggesting the gap matters less at the operational scale than at the diagnostic scale.

### 3.4 Why LR Underperforms Architecturally

After eight systematic variants, F1 plateaued at approximately 0.165 on per-cell metrics — well below the thesis target of 0.80. The cause is architectural:

**LR can only express additive combinations of features:**
```
P(ignite) = sigmoid(w₁·slope + w₂·building + ... + b)
```

Real fire ignition involves multiplicative interactions. A wooden building (high `material_risk`) on a steep slope (high `slope_risk`) with a burning upwind neighbor (high `wind_weighted_score`) is exponentially more dangerous than any single factor alone. LR cannot encode this without explicit interaction features, and even with seven hand-crafted interactions, the model only marginally improved (+0.002 F1).

This is the **representational limitation** of linear models. Adding `wind_weighted_score` was effective because it brought the most important non-linearity into the input directly, rather than asking LR to discover it.

### 3.5 Spatial Validation

The simulation was validated against the Sitio Santa Maria, Lapu-Lapu City fire of December 12, 2023. Ground truth was constructed by Kent through manual digitization of the burned area in QGIS using post-fire Google satellite imagery (estimated ~90% spatial accuracy), since the Bureau of Fire Protection did not provide official burned-area polygons for any fire in the dataset.

**Procedure:**
1. Load Lapu-Lapu rasters (5489 × 6896 grid, EPSG:32651)
2. Configure ignition point at (606748.78, 1141695.77) — converted from GPS to grid index (892, 2730)
3. Wind set to a representative 10 km/h NW (NOT matched to actual Dec 12 2023 weather — see §5)
4. Load Simulation-LR; run CA simulation for 100 timesteps with `early_stop` enabled
5. Save `final_state.tif` (CA cell states 1–5)
6. Treat states {3, 4, 5} (Ignited, Blazing, Extinguished) as "fire"; compare against the binary ground truth using `spatial_validation/validate_simulation.py`

To isolate the value of each ML feature contribution, two LR variants were validated:
- **Simulation-LR (Stage A):** trained on a feature set including `material_class` (a categorical 0–5 building-material code) but excluding `wind_weighted_score`
- **Simulation-LR (Stage B):** trained on a feature set including `wind_weighted_score` (the directional fire-pressure score) but excluding `material_class`

Both stages use the same 9 base features, the same wind-fixed CA engine, and identical hyperparameters; the only difference is the 10th feature.

**Results:**

| Variant | F1 | Recall (primary) | Precision | AUC-ROC | Jaccard |
|---------|:--:|:----------------:|:---------:|:-------:|:-------:|
| CA only (no ML) | 0.684 | 0.544 | 0.921 | 0.772 | 0.520 |
| Simulation-RF (Kent, broken-wind training data) | 0.785 | 0.963 | 0.663 | 0.981 | 0.647 |
| **Simulation-LR Stage A** (material_class) | **0.602** | **0.915** ✓ | 0.448 | 0.958 | 0.430 |
| **Simulation-LR Stage B** (wind_weighted_score) | **0.607** | **0.925** ✓ | 0.452 | 0.963 | 0.436 |

**Confusion matrices (test cells = 37,852,144 total, 3,312 actual fire cells):**

| Stage | TP | FP | FN | TN |
|-------|:--:|:--:|:--:|:--:|
| Stage A | 3,032 | 3,732 | 280 | 37,845,100 |
| Stage B | 3,064 | 3,721 | 248 | 37,845,111 |

Both Simulation-LR variants pass the proposal's primary recall criterion (≥ 0.80). The F1 score falls below the 0.80 target in both stages, driven by over-prediction (the simulation predicts ~6,800 fire cells vs. the actual 3,312, an overestimate of approximately 2×). Per the proposal's recall-prioritized metric hierarchy, missing real fire cells (false negatives) is more costly than over-predicting (false positives), so this error mode is operationally favorable for an early-warning system.

**Limitation on direct LR vs RF comparison:** Kent's reported Simulation-RF F1 = 0.785 was produced with an RF model trained on the original synthetic dataset that predates the wind kernel fix described in §3.1. Both LR stages above use the wind-fixed dataset, which is physically more realistic but a different distribution. Until Simulation-RF is retrained on the wind-fixed data, the LR–RF comparison in this table is **indicative only**. The CA-only baseline, by contrast, is a fair reference for both LR stages because it does not depend on training data.

### 3.6 Stage A vs Stage B — Per-Cell Findings Do Not Fully Translate Spatially

Per-cell evaluation showed the addition of `wind_weighted_score` improved F1 from 0.133 → 0.165 (+22%, see §3.3). Spatially, the same feature change improved F1 only from 0.602 → 0.607 (+0.005, less than 1%):

| Comparison | Per-cell F1 Δ | Spatial F1 Δ |
|------------|:-------------:|:------------:|
| Stage A → Stage B | – | +0.005 |
| 9 features → 10 features (`wind_weighted_score`) | +0.029 | – (proxied by Stage A → Stage B) |

This is an honest finding worth reporting: per-cell metrics on synthetic data overstate the operational value of a feature addition. Possible explanations:
- The benefit of directional wind information is concentrated in early-spread timesteps when the fire is small and directional pressure is the dominant signal. By the time fire has spread to ~3,000 cells (the size of the validation incident), the directional signal is washed out by simultaneous propagation in many directions
- Both `material_class` and `wind_weighted_score` provide useful but partially redundant information at the spatial scale; either is sufficient to break the 9-feature ceiling, and adding both would likely produce only marginal improvement
- The single-incident validation is fragile to specific ignition geometry and weather; different fires may show larger gaps between Stage A and Stage B

For thesis methodology, we recommend reporting both per-cell and spatial metrics with explicit acknowledgment that they measure different things.

### 3.7 Comparison with CA-only Baseline

The CA-only baseline (F1 = 0.684, Recall = 0.544, Precision = 0.921) demonstrates the contribution of the ML component. Without ML, the rule-based CA is conservative — when it predicts fire, it is correct 92% of the time, but it under-predicts fire extent and misses 46% of actual fire cells. Adding LR (Stage B) raises recall from 0.544 to 0.925 (a 70% relative improvement) at the cost of dropping precision from 0.921 to 0.452 (a 51% relative decrease). Net F1 changes from 0.684 to 0.607.

The ML contribution is therefore **a recall-precision rebalance, not a uniform improvement**: the ML model spreads fire to many more cells, catching almost all real fire but at the cost of warning about cells that the real fire did not reach. For a fire-warning operational context where the cost of missing real fire is high (lives, infrastructure), this is the correct direction of trade. For damage estimation, the CA-only baseline may be preferable because of its higher precision.

## 4. Discussion

### 4.1 Pipeline Fidelity Is a Performance Driver — But the Effect Is Scale-Dependent

The per-cell findings of this investigation suggested a strong methodological lesson: **fidelity between simulation physics and ML input features is the dominant performance lever** for CA-ML hybrid systems. Concretely:
- One physics-grounded feature (`wind_weighted_score`) delivered per-cell F1 +0.029
- 100 hyperparameter trials with cross-validation delivered per-cell F1 +0.002
- Seven hand-engineered interactions delivered per-cell F1 +0.002

Replacing the missing physics-aligned feature was approximately 14× more impactful than tuning, and 14× more impactful than feature engineering at the per-cell level.

**However, the spatial validation in §3.5–§3.6 substantially qualifies this finding:** the same `wind_weighted_score` addition that produced +0.029 F1 per-cell produced only +0.005 F1 against the real fire incident. The two stages — one with `material_class`, one with `wind_weighted_score`, both lacking the other — converged to nearly identical spatial performance.

This leads to a more nuanced methodological principle than we initially proposed:

> **In CA-ML hybrid systems, an information asymmetry between the simulator and the ML model can create a measurable per-cell performance ceiling. However, this ceiling does not always translate to a spatial ceiling against real-world ground truth, particularly for incidents large enough that aggregate spread behavior dominates over single-step directional cues. Researchers should evaluate at both per-cell and spatial scales and report both, rather than relying on per-cell metrics alone to make claims about operational value.**

Per-cell evaluation on synthetic data is a useful diagnostic tool, but spatial validation is the criterion that matters for the use case. We are not aware of prior CA-ML fire spread literature reporting this disparity between scales; we submit this dual-scale evaluation methodology as a contribution of this thesis.

### 4.2 LR's Linear Architecture is the Final Ceiling

Even with the information gap closed, LR's per-cell F1 plateaued at ~0.165 because the model cannot capture multiplicative feature interactions. This is consistent with theoretical expectations: the ignition decision is fundamentally non-linear (a "high slope AND high material_risk AND upwind burning neighbors" rule cannot be expressed as a weighted sum). RF's tree splits handle this naturally, which is why we expect Simulation-RF to outperform Simulation-LR in spatial validation.

This finding does not invalidate LR as a baseline. Rather, it quantifies the cost of the linear assumption, which is part of the thesis contribution.

### 4.3 The Recall–Precision Tradeoff

After the information gap fix, LR's behavior shifted from "guess yes whenever near fire" (recall 0.470, precision 0.077) to "guess yes when wind is pushing fire toward the cell" (recall 0.306, precision 0.110). The model became more selective. This is qualitatively closer to the desired behavior for an early-warning system — false positives have real costs (unnecessary evacuation, alert fatigue), and a model that cries wolf less often is more useful even if it slightly under-warns.

The proposal's recall-primary criterion still pushes us toward RF, which achieves recall 0.963 spatially. But LR's qualitative behavior is preferable for some operational use cases where a higher precision threshold is desired.

## 5. Limitations

1. **Synthetic training data:** the model is trained on data generated by the CA itself, introducing potential circularity. Real fire perimeter data would be preferable but is not available for Lapu-Lapu.

2. **Manually digitized ground truth:** the validation ground truth was constructed by tracing the burned area in QGIS using post-fire Google satellite imagery, since the Bureau of Fire Protection did not provide official burned area polygons. This introduces approximately ±10% spatial uncertainty in the validation metric.

3. **Single-incident validation:** at the time of writing, validation has been performed against one historical fire (Sitio Santa Maria, December 12, 2023). Validating against multiple incidents is required to demonstrate generalization; this is identified as ongoing work that depends on additional ground-truth construction by the RF author (Kent), as BFP-supplied burned-area polygons are unavailable.

4. **Wind not matched to actual weather:** the simulation used a representative wind condition (10 km/h NW) rather than the meteorological conditions actually present on December 12, 2023. Wind speed and direction strongly affect fire spread shape and extent, so this represents a substantive methodological limitation. Future work should reconstruct the actual weather using historical archives such as NASA POWER, PAGASA, or NOAA Global Surface Summary, and re-run the validation under those conditions.

4. **Wind not matched to historical weather:** the simulation used a representative wind condition (10 km/h NW) rather than the actual weather conditions on the day of the validation incident. Historical wind data from sources such as NASA POWER or PAGASA could be incorporated as future work.

5. **Per-cell metrics on susceptible cells only:** the per-cell evaluation (Section 3.2) is computed over cells with at least one burning neighbor, matching the CA engine's susceptibility filter. Aggregate metrics across all cells in the grid would be misleadingly high due to the trivial separation provided by `neighbor_burning_count > 0`.

6. **Architectural ceiling of linear models:** F1 ≈ 0.165 on per-cell evaluation reflects LR's inability to capture feature interactions. This is acknowledged as a limitation of the linear baseline, not a deficiency of the methodology.

---

## 6. Conclusion

This thesis chapter evaluated Logistic Regression as the ignition probability estimator within a CA-based fire spread simulation for Lapu-Lapu City, Cebu. Through a systematic eight-variant per-cell investigation and one spatial validation against a historical fire incident, we have:

1. **Identified and corrected an information gap** in the feature pipeline — adding `wind_weighted_score` improved per-cell F1 by 22% (0.133 → 0.165), which is the largest improvement attributable to any single intervention in this study.

2. **Characterized the LR ceiling** through systematic hyperparameter optimization (Optuna, 100 trials, 5-fold CV) and physics-informed feature engineering (seven interaction terms). Neither intervention meaningfully exceeded the F1 ≈ 0.165 plateau, demonstrating that the limitation is architectural, not procedural.

3. **Validated the simulation–LR system** spatially against the Sitio Santa Maria, Dec 12 2023 fire incident in Lapu-Lapu City. Stage A (LR + material_class) achieved Recall = 0.915 and F1 = 0.602; Stage B (LR + wind_weighted_score) achieved Recall = 0.925 and F1 = 0.607. Both pass the proposal's primary recall target (≥ 0.80). F1 falls below the 0.80 secondary target due to ~2× over-prediction of burn area, an error mode that is operationally favorable for early warning but trades against precision.

The primary contribution of this chapter is methodological: **for CA-ML hybrid simulation systems, the fidelity between simulation physics and ML input features is the dominant performance lever**, exceeding the impact of hyperparameter optimization or feature engineering by an order of magnitude. We submit this as a generalizable principle for future CA-ML hybrid work.
